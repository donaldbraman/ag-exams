# Output Structure and Human Review

The `criminal-law` revision pipeline generates a structured set of outputs for each processed class, organized by stage. This structure is designed to facilitate both automated processing and human pedagogical review.

## Directory Organization

All outputs are stored in `outputs/class-XX/`, where `XX` is the zero-padded class number (e.g., `class-09`).

### 1. Stage-Specific Directories
- `stage-1-topics/`: Contains extracted topics and cluster definitions.
- `stage-2-research/`: Contains research results, citations, and summaries. In the specialized architecture (Stage 2.1), multiple steps like `research-cases`, `research-statutes`, and `research-commentary` all map to this directory thanks to the logic `if "research" in step_name` in `run_class.py`.
- `stage-2-retrieval/`: Contains the full text of retrieved legal authorities.
- `stage-3-editing/`: Contains the final edited case studies and commentary, including corrected versions after the hallucination guard pass.

### 2. File Types
- **`[step-name]-gemini.json`**: The raw output from the Gemini model for a specific step. This includes the LLM response, token usage, and latency. **This is the primary source of truth for edited content.**
- **`REVIEW.md`**: A markdown template generated by `scripts/run_class.py` for each stage.
    - **Purpose**: To provide a human-readable summary of the stage's progress and a gate for approval.
    - **Content**: Includes instructions for the reviewer and, where possible, an embedded summary of the "winner" output (e.g., the final edited text or the refined topic list).
    - **Single-Provider Fix**: In the Gemini-only pipeline, the `multi` library does not always populate the `.winner` attribute used by legacy voting logic. The reporting script (`run_class.py`) was updated with a `get_content` helper that extracts the `Provider.GEMINI` output directly if `.winner` is absent.
    - **Refinement Prioritization (Stage 3)**: For the Editing stage, the reporting logic prioritizes the output of the `refine-edits` step (the Corrected Text) over the raw `edit-case` step. If the refinement step exists and succeeded, its content is shown in the `REVIEW.md` to ensure the human reviewer sees the hallucination-free version.
    - **Mechanism**: The `_generate_stage_reviews` function uses this helper to ensure that `refine-topics`, `refine-research`, and `refine-edits` content is correctly embedded in the markdown review files even without a multi-model vote.
    - **Replacement Options (Stage 2)**: For the Research stage, if the "Precedent-Led" workflow is active, the reporting logic embeds a **Full Replacement Options Report**. This report lists multiple 2015-2025 case candidates grouped by the legacy precedent they are intended to replace. The final reading list chosen by the refiner is presented under the heading **## Selected for Retrieval**, providing the professor with both the menu of options and the system's recommended "winners" for the extraction phase.
    - **Common Point of Confusion (Stage-Specific Visibility)**: The "Replacement Options Report" (the choice-rich menu) is generated specifically for the **Stage 2: Research Review**. It does *not* persist into the **Stage 3: Editing Review**, which is narrowly focused on the line-by-line verification and refinement of the single "selected" case study. Users seeking the earlier menu of 18+ candidates (3+ legacy anchors $\times$ 6 candidates each) should always refer back to the `stage-2-research/REVIEW.md` file.
    - **Stage 3 Content Warning**: If Stage 3 contains only placeholders or the "Hawkins" fallback, it usually means the human reviewer has not yet promoted a selection from the Stage 2 menu to the editing phase.

- **`metadata.json`**: A top-level summary of the entire run, including start/end times, success status, and a list of completed steps.

## Document Format Definitions: Edited Cases vs. Summaries

The user-facing output in `stage-3-editing/REVIEW.md` is designed as a **Hybrid Case-Study Document**, specifically tailored for casebook inclusion. It is NOT merely a student summary.

### 1. The "Edited Case" (Extractive Reasoning)
The core of the Stage 3 output is the **Reasoning** section.
- **Nature**: Extractive and abridged.
- **Fidelity**: The system is instructed to use the judge's original words with ellipses (...) for cuts. It should NOT rewrite judicial reasoning into third-person summaries.
- **Length**: Targeted at ~2,000 words for primary cases.

### 2. The "Structured Summary" (Narrative Context)
To assist with rapid pedagogical review, the document includes structured narrative sections:
- **Facts**: A concise narrative summary of the relevant events.
- **Procedural Posture**: A brief description of the case's path through the courts.
- **Issue & Holding**: Clear, concise framings of the legal question and its resolution.

### 3. Verification criteria
When reviewing a Stage 3 document, the primary quality check is whether the **Reasoning** section maintains the "voice" of the court and provides enough depth for classroom interrogation of the doctrine.
- **Verification Support**: The `editing-verifier` provides an automated JSON audit of the Markdown text. Reviewers should check the `verify-edits-gemini.json` file for any "Failed Sentences"—these are passages where the model deviated from the verbatim text or introduced narrative descriptions where direct quotes were expected.
- **Automated Correction**: The `refine-edits` step uses this JSON audit to surgically correct the text. The final version displayed in `REVIEW.md` is the result of this "refine" pass.
- **Human Decision**: Reviewers should confirm that the `refine-edits` pass was successful and that the resulting text flows naturally. If significant "meaning reversals" or non-verbatim reasoning persist, the document should be flagged for re-editing.

## 4. Manual Verification Script: `test_gemini_only.py`
To verify the integrity of the generated outputs without running the full research pipeline, use the `test_gemini_only.py` script.
- **Usage**: `uv run python scripts/test_gemini_only.py <class_number>`
- **Function**: It performs a series of integrity checks on the generated JSON and Markdown files in `outputs/class-XX/`, ensuring that all required structure keys are present and that the final edited content adheres to the expected format.
- **Role**: This serves as a "quick check" gate before a full pedagogical review of the `REVIEW.md` files.

## Human Review Workflow

The pipeline is designed with "human-in-the-loop" checkpoints:

1. **Gatekeeping**: After each stage, the operator should inspect the `REVIEW.md` and associated JSON files in the relevant stage directory.
2. **Approval**: To proceed with confidence (or for historical record), the reviewer can update the YAML block at the bottom of the `REVIEW.md` file:
   ```yaml
   approved: true
   reviewer: [Name]
   notes: [Observations on quality or needed tweaks]
   ```
3. **Iteration**: If a stage's output is unsatisfactory (e.g., poor topic extraction or hallucinated research), the operator should adjust the agent configurations in `config/agents/` and re-run that specific stage or the entire class.

## Data Provenance

Because the `multi` library supports multiple providers, the filenames include the provider name (e.g., `-gemini`, `-claude`). In the current **Gemini-Only** configuration, only `-gemini` files will contain execution data, while others may contain "skipped" markers or be absent depending on the specific script version used.

## Clean Restart Mandate

To maintain data integrity during active development of the `gemini-revision.yaml` workflow, a **Clean Restart** is often mandatory.

- **Requirement**: Delete the class-specific output directory (e.g., `rm -rf outputs/class-09`) before re-running the pipeline if there have been any changes to:
    1. **Step Names**: Renaming steps in the YAML.
    2. **Dependency Graph**: Changing which steps feed into another.
    3. **Step Addition/Removal**: Adding or removing steps from a stage.
- **Rationale**: The underlying `multi` orchestration engine uses directory structure and existing files to track its state. Stale files with old step identifiers can interfere with new parallel execution paths (as seen in the January 2026 refactor of Stage 2 Research).
- **Observation**: A "nuclear cleanup" (deleting the entire `outputs/class-09` directory) was definitive in resolving logical hangs during the transition to the specialized research architecture.

## Troubleshooting Missing Artifacts

In some scenarios, debug logs may indicate that the LLM is responding (e.g., `DEBUG: parallel_chat called`), but no corresponding JSON files appear in the `stage-X/` directories after long periods of execution.

- **Diagnosis**: This often indicates a silent failure or bottleneck in the **State Persistence** or **Saving logic** within the orchestration script (`run_class.py`), or empty results being returned by the model.
- **Tracing Pattern**: To diagnose, inject print statements directly into the `run_class.py` loop that iterates over step results and saves them to disk.
- **Implementation (January 2026 pattern)**:
    ```python
    # In the loop where step_result.outputs are saved:
    print(f"DEBUG: Saving step output for '{step_name}' (provider: {provider.value}) to {output_file}")
    if not step_result.outputs:
         print(f"DEBUG: Step '{step_name}' outputs are EMPTY!")
    else:
         print(f"DEBUG: Step '{step_name}' outputs keys: {list(step_result.outputs.keys())}")
    ```
- **Monkeypatch Result Inspection**: If the saving logic above is never reached (Stage 1 hang), inject the trace higher up into the Gemini monkeypatch in `run_class.py`:
    ```python
    async def new_run_provider(self, provider: str, *args, **kwargs):
        if provider != "gemini": return "", "skipped", 0, 0
        result = await original_run_provider(self, provider, *args, **kwargs)
        # Check if result is a tuple (Standard multi return format)
        if isinstance(result, tuple) and len(result) > 0:
             content_obj = result[0]
             # Inspect LLM standard content attribute
             if hasattr(content_obj, 'content'):
                  print(f"DEBUG: Gemini raw content: {content_obj.content[:200]}")
        return result
    ```
- **Benefit**: This allows the operator to determine if the issue is a failure to complete the step (empty result), a validation rejection by the `multi` engine, or a failure to route the result to the correct file path (path mapping error).
